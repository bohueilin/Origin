The evidence layer for AI agents

Get your agent through security review — and prove what it did.

Origin enforces runtime policy before an AI agent takes a side effect, routes every tool call through a controlled proxy, requires human approval where policy demands it, and produces a tamper-evident, hash-chained evidence package showing what happened, why it was allowed, and who owned the risk.

The evidence package — how one action is bounded and proven

01Proposal Agent proposes a side effect before it can act.
02Policy verdict allow / cap / require-approval / deny / pause.
03Controlled proxy The only path to a real side effect.
04Approval / block Human gate on risky actions; over-scope is blocked.
05Audit chain Every event hash-chained, tamper-evident.
06Evidence package Review-ready export: what, why, who owned the risk.
enforce / plan held / approval verified / recorded red = blocked · gray = simulated

The 90-second evidence loop

Agent proposes→ Policy gate→ Verdict→ Tool-call proxy→ Human approval→ Bounded action→ Audit digest

Watch one workflow run end to end. The proxy is the only path to a side effect. Flip one input and Origin blocks an over-scope action — and records the block. Side effects are sandboxed; any real-money path stays human-created and human-gated, so the agent never touches live money.

Who it's for

The owner of a high-consequence AI agent that's blocked in security review because no one can bound the blast radius or prove what it did.

  • Agents that touch production, internal tools, infrastructure, code, PII, or money-adjacent workflows.
  • First wedge: internal-ops / production-access agents.

What the evidence package contains

  • Each proposal, verdict, approval, action, block, and exception
  • The tool-call proxy events (the only side-effect path)
  • Who owned the risk, and when they approved
  • A hash-chained, tamper-evident audit digest — replayable and exportable

You're ready if

  • You have a high-consequence agent workflow
  • There's a real reviewer and a real blocker
  • There's a tool-call path we can route through a proxy
  • Someone owns approvals for risky actions

Not ready if

  • No high-consequence workflow — the stakes don't warrant it
  • No reviewer or blocker — nothing is actually gating the launch
  • No tool-call path to route through a proxy
  • No owner for approvals on risky actions
  • No willingness to instrument the agent's events

If any of these is true, we'll say so — early.
