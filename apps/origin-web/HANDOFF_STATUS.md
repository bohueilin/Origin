## 0. The one-paragraph pitch
**Origin turns a factory workflow into a deterministic robot-safety gym, tests an agent's
finish / escalate / refuse decisions, and issues a site-specific readiness report before
deployment.** Capability is not permission. The model proposes; a **deterministic BFS oracle is the
sole judge** (no LLM judge, ever). The wedge: site-specific autonomy readiness for factories,
robotics buyers, safety/ops teams, and insurers.

---

## 1. Architecture (what lives where)
This single repo holds three layers:

### a) Marketing home — `src/factorydad/` → `index.html` (`/`)
Frontier-lab landing + benchmark showcase. Sections: Hero ("A robot brain for every floor"),
**RSL curve** (`RslCurve.tsx`), **Projected improvement** (`RsiClimb.tsx`), **per-model Scorecards**
with tabs (`ScorecardPanel.tsx`), **Cost vs readiness** (`CostReadiness.tsx`, `pricing.ts`),
**Trust/Evidence** (`TrustScope.tsx`). Data comes from `src/factorydad/data.json`, generated by the
Python exporter (single source of truth = Python).

### b) Interactive console — `src/` → `app.html` (`/app.html`)
The product flow (state-based views in `src/App.tsx`, no router):
**capture → understanding → reflect/align → illustrate → preview → results (license report)**,
plus a `showcase` ("Sample report"). Initial view is `capture` (the in-app `landing` view is retired).

### c) Deterministic engine — `src/warehouse.ts` (client) + `hud-env/physical-ai-warehouse/` (Python)
Symbolic warehouse grid, `bfsOracle` (the source of truth), verifier with hard-gated reward, FAR/FRR
confusion matrix, blind baselines, signal extractor. The Python side mirrors this and produces
per-model scorecards + `data.json`.

## 2. Trust boundaries (NON-NEGOTIABLE — preserve in all new work)
- **Determinism is sacred.** Same seed/inputs → same world, labels, scores.
- **The oracle/verifier is the only judge.** No LLM judge anywhere.
- **No model sets its own reward, label, FAR/FRR, license, or readiness tier.**
- Voice, video, uploads, site maps, drawn grids, extracted facts = **descriptive inputs** that
  pre-fill a human-reviewed form. They may define the *environment* (the map/task), but the oracle
  alone computes the *verdict*.
- **Secrets stay server-side.** `VITE_*` may only hold public values (e.g. `VITE_API_BASE_URL`).
  No MiniMax/Nebius/InsForge keys in the client bundle (verified clean on each deploy).

## 5. The model-under-test seam (for when the real model is ready)
Everything is shaped so a real agent slots in as the **student**, with the oracle still the judge:
- **Python harness already supports it:** `runner.py` has `anthropic_answer_fn` and a generic
  `openai_compatible_answer_fn` (used for the real MiniMax run). Add a model → run → it earns its own
  tier from the deterministic oracle. Keys via `.env.local`/env, never argv.
- **Console seam:** `LicenseResults.tsx` shows an "Agent under test" placeholder ("integration
  pending — the reference oracle is the bar to clear"). A real model's actions would be scored by
  `verifyWarehouseRollout` exactly like the baselines, and its tier would render alongside the oracle.
- **Server seam:** `server/nebiusHandler.ts` / `minimaxHandler.ts` already proxy models server-side
  with sanitized inputs; the gym (`/v1`) scores authoritatively.

**Rule when wiring the real model:** it only ever *proposes actions*. The oracle computes the
label/reward/FAR/FRR/tier. Never let the model self-report a score.
