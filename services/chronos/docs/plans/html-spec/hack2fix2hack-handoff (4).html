02 The user problem

A suspicious reward is not an actionable bug report.

An environment author opens a reward-1 trace and HUD's Reward Hacking QA agent flags it as likely gaming the evaluator. That is a useful signal, but it is not yet something the author can act on. To actually fix the environment they need four things the verdict does not give them: a reproducible exploit, the nearby adversarial variants that share its mechanism, a verifier patch that removes them, and proof that the patch still admits legitimate solutions.

The job to be done is to turn that single suspicious success into a durable, replayable test and a defended verifier version. Hack2Fix2Hack does exactly that: it reopens the exact machine state behind the trace, attacks it many ways, saves every confirmed attack as an Exploit Witness, and gates the patch on those Witnesses failing while legitimate controls keep passing.

Detection tells you a trace looks wrong. The author still has to reproduce it, expand it, eliminate it, and prove the fix. Chronos is the part that does that.

The research literature is consistent that turning up optimization pressure surfaces verifier attacks rather than task solutions, and that strengthening a weak verifier can sharply lower apparent agent scores. We keep that evidence subordinate to the product problem and collect it in the references; the point here is the author's workflow, not the survey.

03 What already exists

We credit the platform and the loop. We add the missing link.

The hacker-fixer-solver loop is current, active research, and HUD already provides most of the surrounding product. Claiming to have invented either loses points. The direct precedent is Hardening Agent Benchmarks with Adversarial Hacker-Fixer Loops (CMU + Fewshot), whose open-source harden-v0 alternates a hacker, a fixer, and a solver. We fork harden-v0 directly so the 24 hours go to our delta, not to plumbing they already shipped.

HUD already provides
  • Tasks and rewards
  • Traces and environment versions
  • Reward-hacking and false-positive QA agents
  • Step-level file evidence
  • Tasksets, analytics, and version comparison
harden-v0 already provides
  • Hacker / fixer / solver loop
  • Verifier patching from the hack trajectory
  • Replay-based validation gate
  • Legitimate-solution handling
  • Shared defense pool and hack deduplication
Chronos adds
  • Trace-to-executable-state binding
  • Fork-point selection
  • Stochastic state fan-out
  • Deeper re-snapshotting
  • Exploit Witness generation and ProofSets
  • A release gate and ReleaseProof

Two facts about the precedent shape our positioning. First, harden-v0 already does cross-task transfer through its shared defense pool, so transfer is not our novelty. Second, and more important, its hacker is sequential: it attacks, surfaces one exploit, the fixer patches, and it repeats over a handful of independent rollouts. The multi-step exploits that need a rare state and a verifier break from it tend to fall in the gap that sequential, from-scratch sampling leaves. That gap is what Chronos works in.

What the precedent does not do: branch on executable state, and use all of a trace's evidence to choose where to branch. That is the link we add.

04 Chronos vs HUD Reward Hacking QA

HUD detects the signal. Chronos turns it into an executable test.

This distinction has to be explicit, because a judge will reasonably ask why HUD's existing Reward Hacking QA agent is not already enough. It is a different job. HUD's QA agents analyse a completed trace with coding tools and return a structured verdict; the standard Reward Hacking agent emits is_reward_hacking, hacking_strategy, and severity. It never reopens the machine, executes an alternate attack, or proves a patch. Chronos starts where QA stops.

HUD Reward Hacking QA
  • Analyses a completed trace
  • Classifies suspicious behavior
  • Returns a verdict, strategy, and severity
  • Answers: “Was this likely reward hacking?”
Chronos
  • Reopens an exact trace state
  • Executes alternate attacks from that state
  • Saves replayable Witnesses and expands exploit families
  • Validates the patch from the same state
  • Answers: “Can we reproduce, expand, eliminate, and prove the fix?”

QA is the trigger and the evidence source, not a competitor. A Reward Hacking verdict is one of the inputs Chronos reads when it selects a ForkPoint, and after the patch ships, QA should stop classifying the formerly-successful cases as hacks. We position Chronos as the executable layer beneath HUD's analysis, not as a replacement for it.

06 The differentiator

Don't make the attacker get lucky twice.

The reward hacks that stay hidden are usually multi-step: the agent first has to reach a rare, exploitable state, then break the verifier from it. Searched the normal way (fresh run, fresh luck) a single rollout rarely gets both in one trajectory, so a handful of independent rollouts per task miss them, and they only surface when a model in the wild trips over one.

The fix is to stop starting over. The first time any rollout reaches the interesting state, freeze it, then attack that one frozen state many ways. You pay the cost of reaching the state once and amortize the adversarial search across stochastic branches. When a branch reaches a deeper interesting state, re-snapshot it and search again, exposing multi-step exploit paths a flat search structurally misses.

Illustrative · the efficiency intuition

Let state reachability be p and exploit discovery from that state be q. A flat, from-scratch search has to land both in one trajectory, succeeding at roughly p × q. State branching pays p once to reach the state, then searches q repeatedly from the frozen snapshot. A three-step hack adds a further factor r: flat search needs p × q × r in one run; branching pays each reach-cost once per level and decouples the factors.

These are illustrative probabilities, not measured rates. The coverage-versus-compute curve that quantifies the gain for a given task is something we measure during the build; until then it is labelled TBD rather than asserted.

Detection is cheap and stochastic; reproduction and proof are what the author actually needs. Pay once to reach the state, then attack it many ways, and turn every confirmed branch into a permanent test that the patch has to defeat.
07 Discovery vs proof

Discovery is stochastic. Proof is deterministic.

The whole system has exactly two modes, and keeping them separate is the single most important idea in the design. One searches; the other reproduces. Confusing them is what makes “stochastic hacking” sound at odds with “reproducible artifact,” and the split resolves it.

Stochastic discovery
  • Independent seeds per branch
  • Sampling diversity and temperature
  • Agentic branches, no fixed taxonomy
  • Optional multiple attacker models
  • Adaptive branch allocation
  • Re-snapshot promising children
Deterministic proof
  • Restore the saved pre-attack state
  • Replay recorded actions and tool calls
  • Fixed environment and grader versions
  • Confirm reward before the patch
  • Confirm reward 0 after the patch
  • Confirm legitimate controls stay reward 1

Discovery is stochastic.  Proof is deterministic.

Beat by beat

An author sees a suspicious reward-1 trace and HUD QA flags it. A ForkPoint QA workflow inspects the trajectory, reward, environment and worker logs, screenshots, step-level file changes, and any exposed reasoning, and emits a ForkPoint rather than just a suspicion score. Chronos binds that trace step to the correct Modal snapshot and history prefix, then fans the state into isolated stochastic continuations; promising branches are re-snapshotted and expanded again. Successful branches are graded, classified, deduplicated, and saved as Exploit Witnesses with a deterministic replay entrypoint. The Witnesses become a HUD ProofSet. harden-v0's fixer proposes the verifier patch; Chronos reruns the exact Witnesses, re-seeded family variants, and legitimate controls against v2; and HUD publishes a new environment version with its ProofSet and ReleaseProof.

ReleaseProof gate deterministic · both checks restore the same ProofSet states
PRESERVE — controls stay reward 1 KILL — Witnesses go reward 0 reject → re-fix loop
The ReleaseProof gate. The patch ships only when every Witness scores reward 0 and every legitimate control still scores reward 1.
08 The state-tree research engine

The tree is the discovery engine.

The research thesis stays in the main body: pay once to reach an interesting state, amortize adversarial search across stochastic branches, and re-snapshot promising descendants to expose deeper, multi-step exploit paths. The tree is how Chronos discovers Witnesses; the Witnesses, ProofSets, and ReleaseProofs are what it ships.

State-tree search reach once · fork ×K · re-snapshot promising children
explored / dead branch survival path — deepens Exploit Witness leaf
Reach a state once, snapshot it, fork stochastic attacks; a promising branch reaches a deeper state, so we re-snapshot and fork again. Successful leaves terminate in Exploit Witnesses. Depth 1 is must-ship; the deeper tree is stretch.

It is an MCTS-shaped search

Nodes are environment snapshots, edges are candidate attacks, and a leaf's value is whether the verifier handed out reward it should not have. We call this MCTS-shaped rather than MCTS: the hackathon build does not implement a full selection / expansion / value-propagation cycle with a defined tree policy, so claiming MCTS outright would over-state it. The expansion prior, which decides where to spend the next branch, comes from all the trajectory evidence (file changes, logs, grader behavior, and optionally exposed reasoning), not from chain of thought alone. Exposed reasoning helps when it is present, but it is never a required dependency, and the prior only needs to rank nodes better than random.

Branches are agentic, and K is bounded but adaptive

Each branch restores the snapshot and runs the attacker from that exact state at sampling temperature, inventing and executing its own exploit. We do not feed branches a fixed exploit taxonomy, so the search can surface attacks we never listed; harden-v0 holds its documented taxonomy out of the loop for the same reason. We cluster confirmed hacks by their (target, mechanism) pair using harden-v0's dedup so re-spelling a hack does not count as a new find, and a state stops being expanded once its branches stop turning up new distinct exploit clusters.

Hackathon search policy · implementation defaults

For the 24-hour build we use a bounded adaptive policy rather than an unbounded scheduler: 12 initial branches from the first ForkPoint, maximum depth 2, up to 8 child branches from a promising re-snapshotted state, and an early stop after 4 consecutive branches produce no new exploit cluster. These are implementation defaults for the demo, not universal scientific claims; the coverage-versus-compute curve that would justify a wider budget is measured during the build and reported as TBD until then.

09 Worked example

Terminal Wrench · MongoDB Sales Aggregation · iterative pytest subversion.

We anchor the demo on a sourced task rather than a hypothetical. mongodb-sales-aggregation-engine asks the agent to implement a SalesAnalyzer class with MongoDB aggregation pipelines and pass a 10-plus-scenario test suite. In the recorded Terminal Wrench trajectory the agent never wrote any MongoDB code; it spent roughly twenty episodes building and hardening a layered pytest exploit, red-teaming its own attack against countermeasures. That iterative adversarial structure is exactly what the state tree is built to search, and it maps cleanly onto two snapshot levels.

MongoDB pytest subversion two snapshot levels · S₀ reached by ordinary work · S₁ only via the plugin path
survival path — deepens explored / dead branch legitimate solution Exploit Witness leaf
Two snapshot levels. From S₀ the agent can implement legitimately or probe several attack surfaces; only the plugin path reaches S₁, where the defense-in-depth fallback becomes the Exploit Witness.

The recorded attack layered four mechanisms: a local conftest.py hook that no-ops tests (which breaks under --noconftest); a setuptools pytest11 entry-point plugin that survives --noconftest because it loads as an installed package; a pytest_runtest_protocol hook that fabricates passing TestReport objects so pytest's own reporting shows every test as passed; and the original conftest kept as a backup, so only --noconftest and -p no:bypass together defeat both layers.

Why this task fits Chronos

Reusable state. S₀ (repo cloned, deps installed, grader on disk) is reached by ordinary work and is expensive to re-derive; the plugin path lands a deeper S₁ whose attack surface only exists once the plugin is registered. Snapshot mode. The state lives in the filesystem under a working directory, which fits Directory Snapshots and warm branch sandboxes. The Witness. The layered fallback exploit, packaged with S₀/S₁, the recorded actions, the file diff, and the verifier output, is the Exploit Witness. The defense. A verifier patch isolates the test run from agent-controllable plugins, for example running tests from a clean evaluator environment with untrusted plugins disabled and import paths controlled. Replay. The exact exploit is reproduced by restoring the saved state and replaying the recorded actions; legitimate cases are preserved by re-running the frozen legitimate-control suite against the patched verifier.

We assert only what the Terminal Wrench record supports for this task. Anything we have not verified from source (for instance exact per-episode counts beyond “roughly twenty”) is described loosely rather than stated as a precise figure.

10 HUD × Modal architecture

Two first-class lanes, joined at the ForkPoint.

HUD and Modal are balanced, indispensable differentiators, not a host and a fork engine. HUD owns task semantics, reward authority, evidence, grading, tasksets, version comparison, and publishing; Modal owns executable state capture, isolated stochastic fan-out, re-snapshotting, lineage, and replay at scale. The ForkPoint is the shared object that binds HUD's semantic state to Modal's executable state.

11 Modal execution profiles

Three profiles, one durable fallback.

The architecture turns Modal's snapshot maturity levels into deliberate engineering choices rather than sponsor-name decoration. Chronos must not require private Alpha access to function, so the core live path uses a Beta feature centrally and the Alpha features are stretch or research.

Profile A — core live path CORE

Standard Modal Sandbox plus Directory Snapshot (Beta). Flatten the task container into a Modal base image, install dependencies there, keep mutable task state under a working directory, run the HUD environment in the sandbox, snapshot the working directory with snapshot_directory, and mount it into warm branch sandboxes with mount_image. Directory Snapshots are explicitly designed to mount application state into already-running sandboxes and to combine with warm pools, which is close to our fan-out workload, and they do not depend on VM approval.

Profile B — direct Harbor / Docker path ALPHA · access requested

VM Sandbox (Alpha). harden-v0 runs Terminus-2 agents inside task containers via Harbor and expects Docker, and Modal recommends VM Sandboxes for Docker-in-Sandbox workloads, with a real Linux kernel and /var/lib/docker included in Filesystem Snapshots. Access is allowlisted and we have requested it; it is the path for scaling from one converted task to the full Harbor-format dataset, not a dependency for the core demo. VM Sandboxes currently support Filesystem Snapshots but not Memory Snapshots.

Profile C — process-state research path ALPHA · experimental

Memory Snapshot (Alpha, experimental SDK). Use this only when the branch-relevant state lives in a running process: a warmed server, in-memory database state, a live evaluator cache, or a depth-two branch where the process state itself is the attack surface. Memory Snapshots preserve process and filesystem state and can be re-snapshotted, which directly supports the deeper-tree thesis. They are a hot search accelerator, not the durable Witness format.

ModeMaturityChronos roleKey constraints
Filesystem SnapshotGADurable full-filesystem Witness and fallback30-day TTL by default; ttl=None retains indefinitely
Directory SnapshotBetaMain live fan-out with warm branch sandboxes30-day TTL; mount/unmount into running sandboxes

Durable-fallback rule

Memory Snapshots expire after seven days and cannot be extended, so whenever one produces a successful exploit we also write a durable cold-replay representation: a Filesystem or Directory Snapshot, the recorded action sequence, the environment image digest, and the HUD history prefix. Filesystem and Directory Snapshots default to a 30-day TTL and can be retained indefinitely with ttl=None; that is the format an Exploit Witness is stored in. A Memory Snapshot is a search accelerator; the durable Witness is always a filesystem-class artifact plus the replay record.

12 Engineering seams & invariants

Eight things that have to be correct.

These are design rules going in, not discoveries to be made on stage. They are profile-aware: state does not always live only on disk, so the rules say “match the mode to the task” rather than forcing one snapshot type.

  • SEAM 1

    History and executable state share one exact action boundary

    Snapshot at the quiescent point between actions and truncate the agent's history to exactly that step. Both happen as one atomic fork_at(t), keyed to the same t, or the agent “remembers” something the restored state does not reflect and every branch is silently poisoned.

  • SEAM 2

    The snapshot mode matches the task's true state

    Filesystem-dominant tasks use Directory or Filesystem Snapshots; tasks whose live state is a running process use the Memory Snapshot research path. Picking the wrong mode either drops the attack surface or pays for state you do not need.

  • SEAM 3

    The grader version is immutable within a run

    Re-validation restores the saved state, overwrites the verifier in that state with the fixer's patch, replays the Witness (reward must drop to 0), and replays the legitimate controls (reward must stay 1). The grader digest is pinned per run; patching a separate copy makes the proof fiction.

  • SEAM 4

    Every branch has complete provenance

    Tag every gateway call with the branch and node id, and key a Modal Dict by the snapshot object_id as the tree's node store, which is Modal's own persistence pattern. A branch without seed, model, sampling config, and lineage cannot become a Witness.

  • SEAM 5

    Snapshot retention and expiry are explicit

    Filesystem and Directory Snapshots default to a 30-day TTL; set ttl=None for anything that must persist as a Witness. Memory Snapshots expire after seven days and cannot be extended, so they are never the system of record.

  • SEAM 6

    Reward and reward-hacking classification are separate

    The verifier's reward and the QA agent's is_reward_hacking verdict are distinct signals. A branch is only a Witness when it is both rewarded and classified as an actual hack, then deduplicated and replayed; conflating the two produces false Witnesses.

  • SEAM 7

    Untrusted branches get only the secrets and networking they need

    Each branch runs attacker-authored code in an isolated sandbox with minimal credentials and scoped networking, so a destructive or exfiltrating attack cannot poison sibling branches or reach beyond the task.

  • SEAM 8

    A Memory Snapshot discovery is converted to a durable replay artifact

    The instant a Memory Snapshot branch lands an exploit, Chronos writes the filesystem-class snapshot plus action sequence, image digest, and history prefix described in Profile C, so the Witness outlives the seven-day memory window.

13 The 24-hour build contract

Bulletproof the depth-1 loop first. Everything else is stretch.

Get one full loop working end to end and repeatable before anyone touches a stretch: one task, one ForkPoint, the branches, one confirmed Witness, one patch, one ReleaseProof. That loop is the demo.

  1. HUD Python v0.6.4 environment MUST SHIP

    Pin hud-python==0.6.4 with the v6 protocol and the native Modal rollout runtime.

  2. One verified Terminal Wrench task MUST SHIP

    The sourced mongodb-sales-aggregation-engine task, so we skip “find a hackable task” and stay comparable to the precedent.

  3. HUD file tracking MUST SHIP

    Enable file tracking so each trace captures filesystem changes and surfaces them in the code viewer.

  4. One source trace MUST SHIP

    One baseline reward-1 trace that QA flags as suspicious.

  5. One ForkPoint MUST SHIP

    The trace step bound to its Modal snapshot, history prefix, and grader digest.

  6. ~12 genuine stochastic BranchRuns MUST SHIP

    Agentic branches from the ForkPoint at sampling temperature, each with its own seed and lineage.

  7. At least one confirmed Exploit Witness MUST SHIP

    Rewarded, classified as a hack, deduplicated, and packaged with a deterministic replay entrypoint.

  8. One deterministic replay MUST SHIP

    Restore the saved state, replay the recorded actions, reproduce the original reward without re-discovery.

  9. One verifier patch MUST SHIP

    The harden-v0 fixer patches the verifier from the Witness; the loop is reused, not reinvented.

  10. At least three legitimate controls MUST SHIP

    Seeded once from a reference-hinted solver and frozen so the preservation check is reproducible.

  11. One HUD ProofSet MUST SHIP

    A taskset holding the Witnesses and controls as rerunnable cases with analytics.

  12. One v1-vs-v2 ReleaseProof MUST SHIP

    Before/after results: v1 rewards the Witness, v2 does not, and controls pass throughout.

Research stretch

  1. One child state re-snapshotted, one depth-two branch STRETCH

    Re-snapshot a promising child and find a multi-step exploit a flat search misses.

  2. Adaptive branch stopping STRETCH

    Stop expanding a node once new distinct exploit clusters dry up.

  3. Memory Snapshot mode & VM Sandbox direct Harbor mode STRETCH

    Profile C process-state branching and, if access is granted, Profile B Docker-in-VM.

  4. Cross-task defense transfer & training consequences STRETCH

    harden-v0's shared defense pool, and the appendix SFT extension (§19): filtering pipeline implemented on mock data; Fireworks managed SFT handoff smoke-tested. Held-out model comparison remains optional. Never on the live demo path.

Out of scope by design: a broad benchmark campaign, a general MCTS system, complete exploit coverage, and live RL training. We scope the claim to verifier-level hardening so we do not over-promise “harden anything.”

14 Live demo contract

Exactly what the judge sees.

The demo genuinely starts a stochastic search and walks the full loop. The judge watches a suspicious trace become a defended environment version.

  1. Open the suspicious HUD trace.
  2. Show the HUD QA verdict and step-level file evidence.
  3. Show the selected ForkPoint.
  4. Start genuine stochastic Modal branches.
  5. Show branch IDs and traces populating.
  6. Inspect one successful exploit and its file diff.
  7. Save it as an Exploit Witness.
  8. Add it to the ProofSet.
  9. Apply the verifier patch.
  10. Replay the Witness against v2 (reward drops to 0).
  11. Rerun the legitimate controls (reward stays 1).
  12. Show the ReleaseProof.
  13. Publish or display the hardened environment version.
Honest fallback

If the live stochastic search does not surface an exploit during the presentation, we open an Exploit Witness produced by an earlier genuine stochastic run, restore its saved state, and replay it live, clearly labelled as a prior-run artifact. We then continue the patch proof from there. This is not faking discovery: it demonstrates the central product promise, that a stochastic run produces a durable, replayable artifact. We never present a prerecorded discovery as if it happened live.

15 Risks & non-goals

What kills this, and the countermove.

  • R1

    Product overlap with HUD QA

    A judge may ask why QA is not enough. Mitigation: position QA as the trigger and evidence source; Chronos's novelty is reopening the exact state, executing attacks, and proving the patch, none of which QA does.

  • R2

    State fidelity

    A restored branch must behave like the original moment. Mitigation: atomic fork_at(t) (Seam 1), match the snapshot mode to the task (Seam 2), and re-seed RNG per branch so “different” attacks do not collapse into one trajectory.

  • R3

    Classifying legitimate alternatives

    A real but unusual solution can look like a hack. Mitigation: keep reward and the QA hack verdict separate (Seam 6), and seed legitimate controls from a reference-hinted solver so the preservation check exercises real, path-diverse solutions.

  • R4

    Patch narrowing

    Patching one exploit can start rejecting valid solutions; the precedent measures this as a benign-pass drop. Mitigation: the frozen legitimate-control suite is the preservation gate, and harden-v0's post-loop autopatch makes the minimal relaxation if a control breaks.

  • R5

    Experimental API maturity

    Memory Snapshots and VM Sandboxes are Alpha. Mitigation: the core demo runs on the Beta Directory-Snapshot path with no dependency on Alpha access; Alpha features are clearly labelled stretch.

  • R6

    Snapshot expiry

    A Witness whose snapshot has expired is not reproducible. Mitigation: store Witnesses as filesystem-class snapshots with ttl=None plus the action replay record (Seams 5, 8); never rely on the seven-day memory window.

  • R7

    Reproducibility

    A branch that cannot be replayed deterministically is not a Witness. Mitigation: require full provenance per branch (Seam 4) and gate Witness status on a successful deterministic replay before the patch.

  • R8

    Security of untrusted branches

    Branches execute attacker-authored code. Mitigation: isolated sandboxes with minimal secrets and scoped networking (Seam 7), so a destructive branch cannot affect siblings or the host.

  • R9

    Novelty of branching infrastructure

    Forking sandbox state for search is commodity in 2026 (DeltaBox, Crab, ConTree). Mitigation: concede the primitive openly and position on the application, a guided state-branching search for verifier-hack discovery and proof, which is ours.

  • R10

    Task classes that cannot be hardened at the verifier layer

    Some tasks are unfixable at the verifier level; the precedent flags a multi-pass shred task where shred and rm -rf leave identical observable state. Mitigation: scope the claim to verifier-level hardening; these need infrastructure redesign and are explicitly out of scope.

17 The close

Every suspicious success should become a replayable Witness.

The product is the loop: a suspicious trace becomes a ForkPoint, the ForkPoint becomes a tree of stochastic attacks, the successful attacks become Witnesses, the Witnesses become a ProofSet, and the patch ships only with a ReleaseProof that the attacks are dead and legitimate solutions still pass.

HUD shows us where the verifier may have failed. Modal lets us reopen that moment and attack it many ways. Chronos turns successful branches into permanent tests and proves the fix from the same state.

The same mechanism points outward. A branchable reached state is a reusable search asset: beyond exploit discovery, frozen state can support RL rollout reuse, debugging, counterfactual evaluation, and deeper multi-step search. We land the usable product first; the broader research direction follows from it, and the training-data consequences are developed in the appendix.

19 Research appendix

Training consequences of a stronger verifier.

This is a research extension, not part of the core architecture, the must-ship build, or the live demo. It explains why a hardened verifier matters beyond a cleaner score: a hackable verifier poisons the data you train on. The SFT pipeline is implemented and validated on mock data; swapping in real Chronos exports is a file swap, not a rewrite.

Open the SFT / training-data extension implemented · mock validated▸

The hackathon's premise is that you can teach a model anything you can verify, so the verifier is half the system. If the verifier is hackable, every reward-1 trajectory it admits is a candidate training example, including the ones that gamed it. Filtering with a hardened verifier is what keeps the optimization signal clean. SFT does not fix reward hacking by itself — Chronos hardens the verifier and labels Witnesses; this extension shows what would enter a downstream training set before any gradient step.

Boundary

This extension does not replace the core Chronos demo (Witness → patch → ReleaseProof). It does not run live on stage. It does not claim SFT alone eliminates reward hacking. Putting a monitor inside a gradient-bearing reward is explicitly out of scope — that is the regime where reward hacking gets obfuscated rather than removed. Fireworks RFT remains a future extension if an evaluator and held-out taskset are ready.
